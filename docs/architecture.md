## Route Architecture

All routes are defined in `src/App.jsx`. Protected routes check authentication and role before rendering.

```
PUBLIC ROUTES
/                    → Landing.jsx (marketing + app intro)
/login               → Login.jsx   (unified login, redirects by role)

STUDENT ROUTES (requires auth, role = 'student')
/student/menu        → MenuPage.jsx
/student/checkout    → CheckoutPage.jsx
/student/qr          → QRPage.jsx      ← QR display after payment
/student/history     → HistoryPage.jsx

STAFF ROUTES (requires auth, role = 'staff')
/staff/scanner       → ScannerPage.jsx ← Default staff landing page
/staff/orders        → ActiveOrdersPage.jsx

ADMIN ROUTES (requires auth, role = 'admin')
/admin/dashboard     → DashboardPage.jsx ← Default admin landing page
/admin/menu          → MenuManagerPage.jsx
/admin/stock         → StockPage.jsx
/admin/orders        → OrdersPage.jsx
/admin/fraud         → FraudPage.jsx
```

### Protected Route Logic

```jsx
// Role-based redirect after login
const roleRedirectMap = {
  student: '/student/menu',
  staff:   '/staff/scanner',
  admin:   '/admin/dashboard',
};
```

If a user attempts to access a route outside their role, redirect them to their own default route. Never show a 404 for role mismatches — redirect silently.

## Data Flow Architecture

### Order Placement Flow
```
Student selects items
    → useCart.js (local state)
    → Checkout page confirms cart
    → stockLock.js runs: decrements stock in DB for each item
    → Demo payment screen (simulated)
    → On success: useOrders.js inserts order + order_items rows
    → qrTokens.js generates encrypted token
    → Token stored in orders.qr_token column
    → QRPage renders QR code from token
    → Stock remains locked
```

### QR Verification Flow
```
Staff opens ScannerPage
    → html5-qrcode reads camera
    → On scan: qrTokens.js decrypts token
    → useQR.js queries orders table by token
    → Validation: status must be 'paid', qr_scanned_count must be 0
    → If valid: display order details, enable Deliver button
    → On deliver: update status → 'collected', increment qr_scanned_count
    → If invalid: fraudDetection.js logs entry to fraud_logs
```

### Stock Lock Flow
```
Item added to cart → stock_quantity decremented (DB)
Payment fails      → stock_quantity restored (DB)
Order cancelled    → stock_quantity restored (DB)
Order collected    → stock_quantity remains decremented (final state)
```

## System Boundaries

| Boundary | Rule |
|---|---|
| Frontend ↔ Supabase | Only anon key used. Service key never in frontend. |
| Student data | Students can only read/write their own rows (RLS enforced). |
| Staff data | Staff can read all paid orders, update status only. Cannot touch menu or stock. |
| Admin data | Admin has full access to all tables. |
| QR tokens | Encrypted before storage. Never store raw order ID as QR content. |
| Payment | Demo only. No real payment gateway integration in v1. |
